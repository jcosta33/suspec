Swarm framework — visual reference

Five views of how Swarm conditions an agent before it starts work: the pipeline, the recommended routing, the persona model, the skill catalogue, and the session lifecycle. These diagrams are the picture; the prose lives in docs/ — start with docs/concepts/ and docs/reference/flow-graph.md.

1. The conditioning pipeline

A source document suggests a task type, which suggests a persona, which attaches skills and verification gates, which scaffold the conditioned task file the agent reads first.

The routing is recommended, not enforced. A launcher may apply it deterministically when it scaffolds the file; in-session, each skill and persona self-activates on its directive description. The agent self-assesses and re-routes when the work does not match the suggested default, recording the divergence in ## Decisions.

2. Recommended document → task → persona routing

Each source document suggests one default task type and one default persona — a launcher may apply this routing deterministically.

These are the four core routes; the full table covers 18 task types. A missing source document suggests an authoring task that produces the missing doc. The human or launcher may name a different task type when they know something the document does not say.

A persona is a mindset, not a role. The standalone skills load when the work is role-shaped; the carried mindsets arrive with their workflow skill. The catalogue stays at 13 mindsets even though projects may add overlay personas of their own.

Quality gates surface inside whatever task is in play. Authoring skills are one per kind of work — some produce a source doc, some produce code under a discipline. Install only the skills your work needs; persistent project context lives in AGENTS.md, not in a resident skill.

5. The session lifecycle

A scaffolded task becomes active, passes a self-review hard gate, and reaches one terminal state — done, kicked-back, or abandoned. 

stateDiagram-v2
    [*] --> Scaffolded: launcher creates conditioned task file
    Scaffolded --> Active: agent reads file, adopts persona
    Active --> Active: work proceeds, gates fire
    Active --> SelfReview: agent declares work complete
    SelfReview --> Active: hard gate fails, claim unsupported
    SelfReview --> Done: hard gate passes, findings promoted
    SelfReview --> KickedBack: Skeptic review fails
    Active --> Abandoned: blocker is unresolvable
    Done --> [*]: worktree closed
    KickedBack --> [*]: handed off as a new task
    Abandoned --> [*]: documented and closed

The self-review is a hard gate: it refuses incomplete answers and unsupported claims, so an agent cannot declare done by assertion. Task files are gitignored; durable findings migrate upstream before the worktree is deleted.

Authoritative catalogue and rationale live in docs/ — docs/concepts/02-conditioning-pipeline.md, docs/reference/flow-graph.md, docs/personas/, and docs/skills/.
